Overview and Features

Learning databases can be daunting, especially non-intuitive SQL queries and concepts. Confusing and unintuitive database frontends just make matters worse, and are not helpful for beginners. These factors cause many to be put off when faced with the task of learning and implementing a Database Management System (DBMS) such as MySQL, PostgreSQL or MariaDB, instead falling back to non-ideal data storage methods like JSON or CSV files.

OurSQL changes the status quo. Databases are no longer humongous monsters that are tough to learn and keep up with, instead they are tools which provide powerful data handling methods which aid in and support your application. With an interface that is self-explanatory and beautiful to look at as well as features that cater to both the power user and beginner, integrating with DBMSs are no longer a chore, but a pleasure.

OurSQL’s primary aims are highlighted below.

Beautiful Interface

Other graphical database frontends are a pain to look at, and often have an unintuitive and messy interface. One of the main selling points of OurSQL is its beautiful and intuitive interface, which makes working with your database a breeze.

High Performance

Database software should accelerate your database experience, and not slow you down. Many other database frontends (glances at MongoDB Compass) have unacceptably slow performance, with the UI lagging even during simple operations like connecting to a database. OurSQL uses threads and loading statuses effectively to keep the user interface smooth at all times, and keep the user updated with the progress of tasks.

Database Interoperability

OurSQL should work across most SQL databases, so you’ll have a unified application for all your SQL database needs. Switching across multiple clients when moving from one database to the next is needless context switching, which impacts productivity. OurSQL is a centralised application that stores all your database connection parameters and credentials, and allows interfacing graphically with multiple different SQL-based databases. So you can condense all your database management softwares into one.

User Guide

Here you will learn everything you need to get started and scale with OurSQL. This guide is written to be both informative for the experienced user, and provide step-by-step instructions for the novice. Although OurSQL’s interface is highly self-explanatory, this section is here for completeness.

Splash Screen

A splash screen, animated with spring interpolations, displays the app’s icon, title, description and author on launch. The striking animations are short yet beautiful, and will set you in the right mood for using SQL databases every time you launch OurSQL.

1. Cluster Management

Adding a Cluster

Firstly, you’ll need to add a database cluster, which defines a connection to a DBMS. To do so, simply click on the “Add Cluster” button below the cluster list, which summons a dialog where you’ll find:

  1. A sidebar to specify your DBMS type (support subject to change)
  1. Connection parameter fields
  1. Authentication options:
    1. User & Password - Use specified username and password credentials to authenticate
    1. No auth - Connect without any authentication
  1. Credential storage options: (only applicable if user & password authentication is selected)
    1. Encrypted - Encrypt database credentials with a specified password
    1. Plaintext - Store database credentials as plaintext (not recommended)
    1. Don’t store - Prompt for credentials every connection (password is never persisted)
  1. Constructed URI - Actual URI used for connection
  1. Test Connection button - Open a test connection to verify if all parameters and credentials are valid
  1. OK button - Adds and persists this cluster’s configuration

Once you’re done configuring the connection parameters, use the “Test Connection” button to check that a database connection can be successfully established. If you see the dialog similar to the one on the left, all is good, go ahead and hit the “OK” button to add the cluster. If you encounter an error, you’re not alone! Refer to the Appendix for in-depth troubleshooting steps.

Removing Clusters

Made a mistake or no longer need a cluster? Simply right-click any cluster in the sidebar to reveal a menu allowing you to remove it, among other actions. A confirmation dialog will appear to confirm that action. Please note that credentials and connection parameters will be erased permanently from storage, and cannot be undone.

Click on any cluster on the sidebar to view a list of tables in that database. A password dialog (shown below) would appear if a password is required to retrieve connection credentials.

2. Table Actions

Above the table, is a bar packed with table actions.

Running SQL Statements

To run an SQL statement, enter it into the query field and click the run button or hit Enter to execute it. If the statement executed successfully, one of the dialogs below will appear, depending if any results were returned.

Table Action Buttons

Action buttons below the statement field allow you to do a variety of table-related operations. Hovering over any button will reveal a tooltip describing the button’s action. A short summary of each button’s purpose is indicated below, starting from the leftmost button.

Add Column

Add a new column to the current table. You’ll be prompted for the new column’s name and type before it is created.

Add Row

Insert a new row with default values into the current table. (By default, a column’s default value is null, although that can be changed with SQL statements)

Delete Row

Delete (drop) the current selected row from the table. A confirmation dialog will appear before the row is dropped as this is a destructive operation. This button will only be enabled when a row is selected.

Refresh Data

Refresh the current table, fetching a completely fresh copy of it from the database. Some operations might also automatically refresh the table.

Drop Table

Drop the current table from the database. A confirmation dialog will appear before continuing since this is a destructive operation. This cannot be undone.

3. Table Data

The rows in the selected table are displayed neatly in a table as shown below.

Double-click on any cell to edit that cell. A text field appears that allows you to enter the new value of the cell, and clicking anywhere outside the cell or pressing Enter will persist that change in the database. Clearing the cell updates the cell with a null value. The input is validated, and invalid input will be rejected (depending on the column type)

The job status area displays the current or completed job’s status in a clear manner, and catches your attention when required. During a job, the background of the bar would turn orange for the duration of the job. When the task completes, the bar would turn green if it was a success or red if it failed, and go back to its original background (dark grey) after 5 seconds.

MVC Model

Models

Cluster

A class to represent a cluster, stores connection parameters for a single database. Decodable and encodable to facilitate persistence in Java Preferences.

SQL Type Containers

A generic abstract class (Container<?>) was used as the base type of SQL type container classes. These are immutable containers that hold one SQL value of their respective types, and includes methods to validate and escape user input. Various containers extend the base Container class and provide implementations specific to a single or a range of SQL types. The table value validators will call functions of the respective container type for the column to validate input, and prevent inputting of invalid text. As with Codable, extensive tests test every method of all the containers.

Specific DB Utils

An interface (DBMSUtils) was used to allow implementation of various utilities specific to a certain DBMS. For example, to retrieve the connection URI scheme. A helper class (BuiltInDBs) is used to vend the appropriate DBMS’s utils object based on a DBMS identifier.

Views

FXML files were used as views in complex windows, while pure-code was used where it was determined its additional flexibility was beneficial to the component (such as in Dialogs, Alerts etc.). FXML views are found under the ./view/ directory in resources, while smaller pure-code components are found in the com.cryptoalgo.oursql.component package.

Controllers

The backbone of OurSQL, the V in the MVC

Home - Main home screen controller

A variation of the MVVM model is employed, with a view model (HomeViewModel) to offload some logic-heavy operations from the controller. This helps keep the controller from becoming a Massive View Controller. Threading was heavily employed to ensure longer-running operations like SQL statement execution do not block the UI thread. The bulk of the interface logic is contained within this controller and its view model.

AddClusterDialog - Add Cluster flow controller

Validates cluster parameters with Regex and handles adding clusters. Allows creating a test connection to check for validity of connection parameters (threaded, of course). The connection password is stored encrypted or in plaintext, or not stored at all depending on the user’s choice.

Codable

Summary

Interfaces which allow serialisation and deserialisation of classes. Classes contained herein are completely self-sufficient and can be simply copied into your project to be used. Heavily inspired by Swift’s Codable interfaces, but unfortunately slightly crippled by the lack of Java equivalents for certain Swift features.

Classes that conform to Decodable and Encodable can be serialised and deserialised by any Encoder and Decoders.

This class was necessary as many

A built-in implementation that de/serialises values through Java’s Preferences class is available for persisting Clusters to storage in OurSQL. Extensive unit tests were written for the Codable system to test them isolated from the UI.

Implementation (Custom Encoder/Decoders)

Encoder/Decoder (Interface, Generics)

  • Interfaces were used to define the Encoder/Decoder API

KeyedEncodingContainer/KeyedDecodingContainer (Interface)

  • Provides methods for encoding/decoding various types (currently supports String, Integer and Boolean) of data, to be provided by an Encoder/Decoder when requested by the object being encoded/decoded
  • Interfaces were used to allow implementations to provide their own encoding/decoding methods for the various types

More types of containers (e.g. SingleValueContainer, UnkeyedContainer) can be implemented in the future to increase the versatility of this system.

Implementation ((De)serialisable Classes)

Classes that would like to be deserialised/serialised can extend/implement Decodable and Encodable respectively. Any Decoder can decode and create an instance of a Decodable class and the contents of an Encodable class can be encoded by any Encoder, which is the beauty of the Codable system.

  • Encodable is an interface to allow any class to implement it and use the Codable system.
  • Due to limitations in Java’s interfaces, namely that interfaces cannot specify mandatory constructors, Decodable is an abstract class although an interface would very much fit its use case better. Decodable classes are required to have a constructor that only takes a Decoder as an argument, which initialises all required fields via Decoder methods.

JDBC Drivers

Drivers conforming to the JDBC standard provide a unified interface for SQL-based DBMSs, allowing operations to be carried out with a single JDBC API regardless of the DBMS in use.

JDBC drivers were crucial in the architecture of OurSQL, providing the backbone for interfacing with various SQL databases with a single codebase. Currently, only MySQL and PostgreSQL JDBC drivers are linked into the application, but adding another driver down the road is as simple as adding a dependancy. Drivers allow execution of arbitrary SQL statements, retrieving database info and more, with a unified database not dependant on the specific DBMS.

SecretsStore

A store, for storing secrets

Stores secrets in an encrypted form with a provided password. A plaintext storage option is also provided. Helps abstract away crypto and persisting operations from the UI code, and to allow for unit testing of storage and retrieval operations.

Testing Methodology

Testing methods detailed below are stated in (mainly) chronological order.

Unit Testing

Tests of crucial components were written using the JUnit test framework. The unit test files are present in the ./src/main/test directory. During development and after every large change, the tests are run to ensure no existing functionality was broken. They also allow isolated debugging of issues pertaining to those components, without interference by UI code. Tests for almost every “backend” component were written to catch regressions early. A badge displays the latest test status, run by GitHub actions on every commit.

Screenshots of some tests are included for reference below.

White-box Testing

The application was subject to rigorous manual testing during development (by the developer) to weed out UI bugs. Multiple instances of databases were tested to ensure compatibility with all possible combination of DBMS types. This form of testing took place every run to test a subset of features that were being worked on. One of the many examples of bugs found was a wrong function call in the TextField change formatter, which caused the validator to only validate the added string in the change instead of the whole string in the TextField.

As an added safety net, the whole project was version-controlled by Git and pushed to GitHub. Commits were made as often as possible (to date there are 86+ commits). This both acts as a backup, and allows me to easily rollback breaking modifications.

Finally, the last stage of testing was to ensure it worked cross platform. This part was the most tedious, as there were issues that only surfaced when ran on Windows, and OurSQL was developed solely on macOS until now. Git allowed quick syncing of changes between both machines, and the modifications to fix windows issues were ran on macOS to ensure they didn't break existing working features. An example of a bug found during this stage was a difference the way windows handles paths, using \ (backslashes) instead of / (forward slashes) to indicate a path component. This broke connection URI construction, and had to be fixed by forcing the use of a forward slash at all times.

Stress Testing

After major features were completed (such as loading of tables/rows, adding clusters, adding rows/columns and adding rows), they were subject to random invalid input and stress tested with unforgiving behaviour. A few bugs, mainly pertaining to race conditions, were found through this method. For example, a race condition when setting statuses caused “null” to be displayed as the completed task when statuses are changed quickly (such as on a localhost database where latency is close to zero). Another example is a bug that caused connections not to be closed immediately after they are not needed, leading to a build-up of stale connections when new tables are selected in quick succession that eventually caused further connections to the database to fail with a “too many open connections” error.

Black-box Testing

The application was demonstrated to classmates at various stages of development to simulate a normal use case scenario. Any bugs were then found and weeded out. Fortunately, no bugs were found at this stage of testing, since it usually took place after both white-box testing and stress testing by the developer. Finally, prior to the submission of the project, the final JAR file was sent out to several friends to ensure it really worked cross-platform. It was also tested on my Mac and Windows laptops.

Reflection

Obstacles Faced

Suffice to say, my experience with JavaFX and Java was a far cry from enjoyable. I had to reengineer code multiple times to work around problems I had not anticipated, and make do with producing a less

Although I strive to ship a quality product with high standards of code, I had to reduce my expectations as the deadline approached. This has unfortunately led to some compromises, in the form of janky bug fixes and a departure from the beautiful and maintainable code I was used to working with in other projects. Features such as MongoDB support was removed after they were deemed too ambitious, and animations were notoriously tough to work with in JavaFX, so I had to make do with a less animation-heavy experience.

Another obstacle faced was targeting lower Java versions when building the JAR, and bundling in the correct libraries that would truly make it cross platform. By default, only the JavaFX graphics pipeline for the current platform (and arch) would be linked into and included with the compiled product. As I primarily develop on a Mac (an Apple Silicon one in fact), the JAR compiled on that platform would only be able to run on macOS and the aarch64 architecture. It took longer than it should have to fix this problem, but in the end, I could produce a JAR that was truly “cross platform” that ran on Windows (x64) and macOS (aarch64 and x64).

Translating all the app’s 100+ strings to Chinese was also a challenge for me, as my Chinese was fairly weak. I had to rely on translation services like DeepL translator to obtain a translation, then cleaned up the translation using my limited Chinese knowledge.

Finally, prior to this project, I had absolutely zero SQL experience. All my database experience was in using NoSQL databases, such as MongoDB, which have a completely different API. I had to juggle between learning the syntax of SQL statements and writing code that interfaced with the database with those statements, which was no easy task. Fortunately, learning SQL statements was not as daunting of a task as I had anticipated, and I wrote abstractions to make database logic separate from UI management to reduce the time I had to spend working with raw SQL statements.

Learning Points

Although developing with JavaFX wasn’t desirable, it gave me the valuable experience of native cross platform development. I have also learnt how to curb my expectations of the standard of my product, allowing me to ship an acceptable product within the stipulated time, which is very important in a working environment. Since I have a habit of seeking perfection in my projects, it was a rather difficult lesson to learn.

Next, I learnt that I shouldn’t wait till the last minute to test compatibility of the app on Windows. I was fortunate enough to own both a powerful Windows laptop and a powerful Mac which could build the whole app really quickly, which allowed me to quickly iterate and solve the problems. In the future, I should periodically test my app on other platforms so I can fix these inconsistencies earlier in the development cycle.

Future Works

Due to limited time, a few features had to be scaled down. Namely, the help feature, that was initially planned to show interactive SQL statement tutorials with blanks that could be filled in by the user to construct an SQL statement, was omitted due to a lack of development time.

A bulk data import feature with CSV/JSON files was also planned, but ultimately wasn’t implemented due to the lack of time.

Lastly, a very minimal set of SQL types are supported due to the high amount of time required to implement and test every container type. I’d also like to add more advanced SQL features such as managing column constraints, creating and monitoring indexes, and supporting more SQL operations such as dropping columns.
